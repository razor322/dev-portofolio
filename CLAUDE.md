### Feature Modules (src/features/)

Domain-driven structure. Each feature owns its components, data, and types.

- `doc/` â€” Centralized content system. All MDX content lives in `src/features/doc/content/`. Data layer at `src/features/doc/data/documents.ts` exports `getAllDocs`, `getDocBySlug`, `getDocsByCategory`. Blog posts vs component docs are distinguished by the `category` frontmatter field.
- `blog/` â€” Blog UI only; data comes from `features/doc`
- `portfolio/` â€” Portfolio data, components, and types

### Component Registry (src/registry/)

The project doubles as a shadcn component registry (namespace `@ncdai`). Registry types and their definition files:

| Type                 | File                                   |
| -------------------- | -------------------------------------- |
| `registry:component` | `src/registry/components/_registry.ts` |
| `registry:hook`      | `src/registry/hooks/_registry.ts`      |
| `registry:block`     | `src/registry/blocks/_registry.ts`     |
| `registry:example`   | `src/registry/examples/_registry.ts`   |
| `registry:lib`       | `src/registry/lib/_registry.ts`        |

**NEVER EDIT** auto-generated files: `src/registry/__index__.tsx`, `src/registry/registry.autogenerated.json`, `public/r/*.json`

#### Adding a New Registry Component

1. Create component in `src/registry/[name]/`
2. Register in the appropriate `_registry.ts` file
3. Create example in `src/registry/examples/`
4. Run `pnpm registry:build`
5. Add docs MDX in `src/features/doc/content/` with `category: "components"`

## Coding Conventions

- **TypeScript strict mode**; explicit types when necessary
- **kebab-case** file naming
- **No emojis** in code, comments, or commit messages
- Comments only for "why", not "what"
- **Tailwind CSS v4** syntax; always support dark/light modes
- **Path alias**: `@/*` maps to `./src/*`

### Import Order (enforced by Prettier)

1. `react` / `react/*`
2. `next` / `next/*`
3. Third-party modules
4. `@/types/*`, `@/config/*`, `@/lib/*`, `@/hooks/*`
5. `@/components/ui/*`, `@/components/*`
6. `@/registry/*`, `@/styles/*`, `@/app/*`, `@/features/*`
7. Relative imports

### Prettier Config

No semicolons, double quotes, import sorting via `@ianvs/prettier-plugin-sort-imports`, Tailwind class sorting via `prettier-plugin-tailwindcss`.

## Component Documentation

Component docs are MDX files in `src/features/doc/content/`. The `description` field must match between MDX frontmatter and the registry `_registry.ts` entry.

Doc section order: Preview â†’ Features â†’ Installation â†’ Usage â†’ Composition â†’ API Reference â†’ Examples â†’ Credits â†’ References. Only include sections that are relevant.

- **Description**: One short sentence, starts with verb or noun, no "A/An", no implementation details, ~80 chars max
- **Features**: 2-4 bullets, only for non-obvious capabilities; skip if self-explanatory
- **Composition**: Only for compound/composable components; use box-drawing characters in a `text` code block
- **Credits**: Who inspired it; link text is author name/handle only, URL points to exact source
- **References**: Technical reading for implementation; keep links specific and high-value
